# ADR 0090 — 운영 대시보드의 관리자 API 인증 누락 방지

- **Status**: Accepted — 수정 및 운영 반영
- **Date**: 2026-07-16
- **Scope**: `app/admin/page.tsx`, `app/admin/page.test.tsx`, `GET /api/admin/resilience`
- **연관**: ssuMCP `JwtAuthFilter`, `@AuthUser`, `SSUAI_ADMIN_STUDENT_IDS`

## 장애 맥락

로그인한 운영자가 `/admin`에 진입하면 화면은 정상적으로 렌더링됐지만 Resilience4j 서킷 브레이커
조회만 `API 호출 실패: HTTP 401`로 끝났다. 기대 동작은 허용 목록에 든 로그인 사용자가 실시간
상태를 보고 새로고침과 30초 폴링을 계속 사용하는 것이다. 실제 동작은 모든 조회가 인증되지 않은
요청으로 처리되어 대시보드 핵심 데이터가 비어 있었다.

재현 절차는 다음과 같다.

1. 유효한 ssuAI 세션으로 로그인한다.
2. `/admin`에 진입한다.
3. 브라우저 네트워크 기록에서 `/api/admin/resilience` 요청을 확인한다.
4. 요청에 `Authorization` 헤더가 없고 응답이 HTTP 401인지 확인한다.

영향 범위는 운영 대시보드 조회로 한정됐다. 일반 사용자 화면과 백엔드의 서킷 브레이커 자체는
정상 동작했지만, 장애 시 가장 먼저 확인해야 할 운영 관측 화면을 사용할 수 없었다.

## 가설과 증거

- **백엔드 또는 외부 시스템 장애**: 기각했다. 응답이 연결 실패나 5xx가 아니라 인증 계층의
  401이었고, 같은 엔드포인트는 Bearer access JWT가 있을 때만 `JwtAuthFilter`가 사용자 식별자를
  설정하도록 설계되어 있다.
- **클라이언트의 로그인 확인만으로 API도 인증됨**: 기각했다. `isAuthenticated`는 화면 접근을
  제어할 뿐 `fetch()` 요청에 자격 증명을 자동으로 복사하지 않는다.
- **Next.js rewrite가 인증을 주입함**: 기각했다. `/api/admin/resilience` rewrite는 요청을
  ssuMCP로 전달하지만 브라우저 메모리에 있는 access JWT를 알거나 `Authorization`으로 변환하지
  않는다.

## 근본 원인

관리자 페이지는 인증 상태를 확인한 뒤 데이터를 요청했지만, 실제 `fetch()`에는 현재 access JWT를
전달하지 않았다. 따라서 ssuMCP의 `JwtAuthFilter`가 인증 주체를 만들지 못했고 `@AuthUser` 인자
해석 단계에서 401을 반환했다. 화면의 로그인 게이트와 API 호출의 인증 계약을 같은 것으로 취급한
것이 설계 결함이었다.

## 검토한 대안

### A. 관리자 엔드포인트를 공개 조회로 변경

기각했다. 서킷 브레이커 이름과 상태는 운영 정보이며, 기존 사용자 인증과 관리자 허용 목록을
우회할 이유가 없다. 화면 오류를 없애기 위해 서버 보안 경계를 약화하면 안 된다.

### B. Next.js 서버 계층에서 고정 관리자 자격 증명 주입

기각했다. 실제 로그인 사용자와 무관한 공유 자격 증명이 생기고, 사용자별 감사와 권한 회수가
불가능해진다. 배포 비밀 관리도 추가되어 문제보다 큰 운영 부담을 만든다.

### C. 현재 사용자의 access JWT를 요청마다 전달 (채택)

`useSaintAuth()`가 제공하는 현재 access token을 API 헬퍼에 명시적으로 넘기고 모든 초기 조회,
수동 새로고침, 30초 폴링에 `Authorization: Bearer <token>`을 붙인다. 토큰 회전 시 effect가 다시
실행되어 새 토큰으로 재조회한다. 이 방식은 ssuMCP의 기존 인증·관리자 허용 목록 계약을 그대로
사용하며 별도 자격 증명을 만들지 않는다.

## 결정과 구현

1. 관리자 API 헬퍼는 access token 없이는 호출되지 않으며, 모든 요청에 Bearer 헤더를 붙인다.
2. 요청 세대 번호를 두어 토큰 교체 이전의 느린 응답이나 수동 새로고침 응답이 최신 상태를
   덮어쓰지 못하게 한다.
3. 성공한 응답은 이전 오류를 지운다.
4. 401과 403은 초기 조회와 폴링 모두 사용자에게 표시하고, 이미 보이던 보호 데이터와 마지막
   갱신 시각을 즉시 지운다. 권한이 회수된 뒤 운영 정보가 화면에 남지 않게 하기 위한 조치다.
5. 일시적인 폴링 실패는 현재 데이터를 유지하고 다음 주기에 재시도해 순간적인 네트워크 오류로
   화면이 흔들리지 않게 한다.

## 검증과 회귀 방지

컴포넌트 테스트는 다음 경계를 고정한다.

- 최초 조회와 수동 새로고침에 현재 Bearer 토큰이 포함된다.
- access token 회전 후 새 토큰으로 재조회하고 이전 요청의 늦은 응답은 무시한다.
- 401 뒤 토큰 갱신 요청이 성공하면 오류가 사라진다.
- 폴링 중 403이면 보호 데이터와 마지막 갱신 시각이 제거된다.
- 로그아웃 상태에서는 관리자 API를 호출하지 않는다.

로컬에서는 ESLint, TypeScript 검사, Vitest 160개, Next.js production build를 통과했다. PR #243의
CI, gitleaks, Vercel preview를 통과한 뒤 commit `989f6e3`이 `main`에 반영됐고, 동일 커밋의
Vercel production 배포가 성공했다.

## 남은 위험과 운영 확인

- 인증이 정상이어도 `SSUAI_ADMIN_STUDENT_IDS`에 현재 운영자 학번이 없으면 의도대로 403이 난다.
  401은 토큰 전달 문제, 403은 관리자 허용 목록 문제로 구분해 대응한다.
- access token은 브라우저 메모리에만 두며 로그, 오류 메시지, 테스트 스냅샷에 값을 남기지 않는다.
- 배포 후에는 실제 허용된 운영자 세션으로 `/admin`을 열어 데이터가 표시되고 새로고침과 한 번의
  폴링이 모두 200인지 확인한다.

## 인터뷰에서 설명할 질문

- 로그인된 React 화면에서도 API 요청이 401이 될 수 있는 이유는 무엇인가?
- 토큰 회전과 느린 응답이 겹칠 때 오래된 관리자 데이터가 다시 나타나는 경쟁 조건을 어떻게
  막았는가?
- 폴링의 일시적 실패와 권한 상실(401/403)을 UI에서 다르게 처리한 이유는 무엇인가?
